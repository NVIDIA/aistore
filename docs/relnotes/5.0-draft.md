### Networking: public and intra-cluster networks must be separate

AIS supports three logical networks:

* **Public network**: user-facing API, CLI/SDK traffic, metrics, and external tooling.
* **Intra-cluster control network**: health, metasync, membership, xaction control, and internal control-plane broadcast.
* **Intra-cluster data network**: object movement between AIS nodes, including target-to-target GET/PUT, global rebalance, resilver, get-batch streams, and other data-mover paths.

These are **logical roles**. They may map to separate physical networks, or they
may share physical infrastructure, depending on deployment size and performance
requirements.

Starting with v5.0, AIS requires the public network to be separated from
intra-cluster networking at the HTTP listener level.

In practical terms:

* the public endpoint must not be the same listener as intra-control or intra-data;
* intra-control and intra-data may share the same listener;
* public multi-homing is allowed, but all public addresses remain public-only;
* physical network isolation is strongly recommended for production, but is not required by this rule.

The mandatory rule is about **endpoint separation**, not necessarily
physical NIC separation. A deployment may use fewer physical networks than
logical roles, as long as the public listener is distinct from the intra-cluster
listener.

A v4.x deployment with a collapsed (single-listener) `host_net` will not start on
v5.0. Update `host_net` before upgrading.

**Example: minimal two-listener deployment**

```json
"host_net": {
    "hostname": "10.50.56.200",
    "hostname_intra_control": "10.50.56.205",
    "hostname_intra_data": "10.50.56.205",
    "port": "51081",
    "port_intra_control": "51081",
    "port_intra_data": "51081"
}
```

In this configuration:

* public traffic uses `10.50.56.200:51081`;
* intra-control and intra-data share `10.50.56.205:51081`;
* AIS runs two HTTP listeners and two HTTP servers: one public, one intra-cluster.

> Note: collapsing intra-nets is not recommended for the usual reasons that boil down to different (maybe even opposite) requirements:
> control traffic is low volume and highly latency-sensitive, whereas intra-cluster data requires high bandwidth, large Tx/Rx buffers, Jumbo frames, etc.

### Direct target access is disabled when AuthN or intra-cluster security is enabled

Historically, some deployments allowed users or automation to send object
requests directly to target public endpoints, bypassing proxy mediation.
Starting with v5.0, direct target access is rejected when either of the following
is configured:

* AuthN is enabled;
* `auth.intra_cluster.enabled` is set.

AuthN and intra-cluster request protection secure different boundaries: identity
at the public perimeter, and integrity of internal node-to-node traffic. Both,
however, rely on the same operational invariant - user traffic must pass through
a proxy. For that precise reason, enabling either knob disables direct user
access to target endpoints.

This applies to user-initiated requests arriving directly at a target public
listener. Proxy-mediated requests, including proxy-to-target redirects, continue
to work and are verified internally.

**Operator action**: point clients, scripts, load balancers, and automation at
proxy endpoints (see `AIS_ENDPOINT`).

> Note: this is one of the two v5.0 effects of `auth.intra_cluster.enabled` that
> apply even though signing itself is inactive - see "Configuring intra-cluster
> signing during the v5.0 window" below.

### `auth.cluster_key` => `auth.intra_cluster` config rename

* **What changed**

  * The cluster config section formerly named `auth.cluster_key` has been renamed to `auth.intra_cluster`.
  * The new name is mechanism-neutral. Older releases used a symmetric cluster-key/HMAC mechanism; v5.x moves toward per-node signing.

* **Compatibility**

  * Persisted global configs written by pre-v5.0 nodes are accepted on startup and automatically mapped to `auth.intra_cluster`.
  * No data-format migration is required, and no manual edit of persisted config is needed **for the rename**.

This compatibility applies to persisted cluster configuration loaded during upgrade.

> Note that v5.0 separately requires public/intra-cluster listener separation (see above), which may require a `host_net` change before the cluster will start.

* **Interactive updates: CLI and REST single-field updates**

  Single-field updates resolve the dotted path against the current JSON field
  names. Use the new path:

  ```console
  $ ais config cluster auth.intra_cluster.enabled true
  ```

  The legacy dotted path no longer resolves and is rejected:

  ```console
  $ ais config cluster auth.cluster_key.ttl 1h
  Error: unknown property "auth.cluster_key.ttl"
  ```

### v5.0 bridge release: legacy CSK/HMAC removed

v5.0 is a bridge release between the v4.x symmetric cluster-key mechanism and the
v5.x per-node signing mechanism.

* **What changed**

  * The deprecated symmetric CSK/HMAC intra-cluster signing path is removed in v5.0.
  * v5.0 nodes generate and publish per-node Ed25519 verifying keys.
  * Ed25519 request signing and verification is **not** activated in v5.0.

Intra-cluster protection is enabled in v5.1, which follows shortly after v5.0, once the cluster has crossed the v5.0 bridge and all admitted nodes publish verifying keys.

* **Upgrade sequencing**

  * v5.0 is the required bridge for crossing from v4.x to later v5.x releases.
  * Direct upgrade from v4.x to post-v5.0 releases is not supported.
  * Upgrade the cluster to v5.0 first, then proceed to v5.1.

* **Operational impact**

  * Operators that require protected intra-cluster traffic during the v5.0 bridge window should use isolated intra-cluster networking and/or TLS.

* **Deprecation note**

  * Accepting legacy persisted `auth.cluster_key` configuration on load is intended only for the v5.0 upgrade window.

#### Configuring intra-cluster signing during the v5.0 window

The `auth.intra_cluster.*` section is fully configurable in v5.0. Values are
validated and defaulted when you set them, and the resulting configuration
persists across the upgrade to v5.1. Pre-staging the section during the v5.0
window is supported, and is the recommended sequence: configuration errors
surface while signing is still inactive, rather than at the moment you turn it
on.

```console
$ ais config cluster auth.intra_cluster.enabled true
```

Unset durations are filled in with defaults:

| Property                            | Default | Notes                       |
|-------------------------------------|---------|-----------------------------|
| `auth.intra_cluster.ttl`            | `0s`    | 0 = signatures never expire |
| `auth.intra_cluster.nonce_window`   | `1m`    | must be > 0                 |
| `auth.intra_cluster.rotation_grace` | `1m`    | must be > 0                 |

In v5.0, setting `auth.intra_cluster.enabled`:

* does **not** activate Ed25519 request signing or verification - v5.1 does;
* does **not** re-enable the removed CSK/HMAC mechanism;
* **does** take effect as request-path policy - direct user access to target endpoints is rejected (see "Direct target access is disabled ..." above).

During the rolling upgrade to v5.1, nodes still running v5.0 do not sign. Complete
the upgrade on all nodes before relying on intra-cluster signing.
