## Upgrade Notes

### `auth.cluster_key` => `auth.intra_cluster` (config rename)

- **What changed**
  - The cluster config section formerly named `auth.cluster_key` has been renamed to `auth.intra_cluster`. The new name is mechanism-neutral: v4.x used a symmetric CSK/HMAC mechanism; v5.0+ moves toward per-node signing.

- **Compatibility**
  - Persisted global configs written by pre-v5.0 nodes are accepted on startup and automatically mapped to `auth.intra_cluster`. No data-format migration is required, and no manual edit of persisted config is needed.
  - The legacy CSK/HMAC implementation is retained for the upgrade window to preserve current behavior; it will be removed in a future release once migrations complete.

- **Interactive updates (CLI / REST single-field updates)**
  - Single-field updates (e.g. `ais config cluster ...`) resolve the dotted path against the current JSON field names. Use the new path `auth.intra_cluster.*`.
  - Example - enable intra-cluster signing:
```
    ais config cluster auth.intra_cluster.enabled true
```
  - The legacy dotted path no longer resolves and is rejected outright:
```
    $ ais config cluster auth.cluster_key.ttl 1h
    Error: unknown property "auth.cluster_key.ttl"
```
    Reissue using `auth.intra_cluster.*`. (A malformed value under the *new* name is a separate, ordinary validation error - e.g. `invalid intra_cluster.ttl 3s (expecting >= 1h0m0s)`.)
  - Note: a config *payload* (JSON body or file) that still carries the legacy `cluster_key` section is silently ignored on update - the section is dropped, not migrated, and not flagged. Always use `intra_cluster` in update payloads.

- **Recommended operator actions**
  - Persisted configs need no action - they are migrated automatically on load.
  - Update automation, scripts, and ad-hoc CLI/REST calls to use `auth.intra_cluster.*`.
  - For any JSON config you apply via *updates* (as opposed to persisted state), search-and-replace the key first: `"cluster_key":` → `"intra_cluster":`.

- **Troubleshooting**
  - `unknown property "auth.cluster_key.*"` on a `set-config` means you're using the legacy dotted path - reissue with `auth.intra_cluster.*`.
  - An `intra_cluster` update supplied as a JSON payload that appears to do nothing likely still named the section `cluster_key`; rename it in the payload.

- **Deprecation note**
  - Accepting legacy persisted configs on load is intended only for the upgrade window. The legacy CSK/HMAC code path and the `auth.cluster_key` name will be removed in a future release; migrate scripts and CI to `auth.intra_cluster` as part of your v5.x rollout.
