## Upgrade Notes

### `auth.cluster_key` => `auth.intra_cluster` config rename

* **What changed**

  * The cluster config section formerly named `auth.cluster_key` has been renamed to `auth.intra_cluster`.
  * The new name is mechanism-neutral. Older releases used a symmetric cluster-key/HMAC mechanism; v5.x moves toward per-node signing.

* **Compatibility**

  * Persisted global configs written by pre-v5.0 nodes are accepted on startup and automatically mapped to `auth.intra_cluster`.
  * No data-format migration is required, and no manual edit of persisted config is needed for cluster startup.
  * This compatibility applies to persisted cluster configuration loaded during upgrade. It does **not** mean that the legacy dotted path remains valid for config updates.

* **Interactive updates: CLI and REST single-field updates**

  * Single-field updates resolve the dotted path against the current JSON field names.
  * Use the new path: `auth.intra_cluster.*`.

  Example:

  ```console
  $ ais config cluster auth.intra_cluster.enabled true
  ```

  The legacy dotted path no longer resolves and is rejected:

  ```console
  $ ais config cluster auth.cluster_key.ttl 1h
  Error: unknown property "auth.cluster_key.ttl"
  ```

  Reissue the command using `auth.intra_cluster.*`.

  A malformed value under the new name is a separate validation error, for example:

  ```console
  Error: invalid intra_cluster.ttl 3s (expecting >= 1h0m0s)
  ```

* **JSON config update payloads**

  * A config update payload that still carries the legacy `cluster_key` section is not migrated as part of the update.
  * Use `intra_cluster` in update payloads.
  * Before applying JSON config updates, replace:

  ```json
  "cluster_key":
  ```

  with:

  ```json
  "intra_cluster":
  ```

* **Recommended operator actions**

  * Persisted configs need no manual action; they are mapped automatically on load.
  * Update automation, scripts, CI, and ad-hoc CLI/REST calls to use `auth.intra_cluster.*`.
  * Update JSON config files used for cluster updates to use `"intra_cluster"`.

* **Troubleshooting**

  * `unknown property "auth.cluster_key.*"` means the command is using the legacy dotted path. Reissue with `auth.intra_cluster.*`.
  * An `intra_cluster` update supplied as a JSON payload that appears to do nothing may still be using the legacy section name `cluster_key`; rename it in the payload.

### v5.0 bridge release: legacy CSK/HMAC removed

v5.0 is a bridge release between the v4.x symmetric cluster-key mechanism and the v5.x per-node signing mechanism.

* **What changed**

  * The deprecated symmetric CSK/HMAC intra-cluster signing path is removed in v5.0.
  * v5.0 nodes generate and publish per-node Ed25519 verifying keys.
  * Ed25519 request signing and verification is not activated in v5.0. It is enabled only in a later v5.x release after the cluster has crossed the v5.0 bridge and all admitted nodes can publish verifying keys.

* **Operational impact**

  * `auth.intra_cluster.enabled` may still appear in configuration for compatibility and future use.
  * In v5.0, enabling `auth.intra_cluster.enabled` does not re-enable the deprecated CSK/HMAC mechanism.
  * Operators that require protected intra-cluster traffic during the v5.0 bridge window should use isolated intra-cluster networking and/or TLS.

* **Upgrade sequencing**

  * v5.0 is the required bridge for crossing from v4.x to later v5.x releases.
  * Direct upgrade from v4.x to post-v5.0 releases is not supported.
  * Upgrade the cluster to v5.0 first, then proceed to later v5.x releases.

* **Deprecation note**

  * Accepting legacy persisted `auth.cluster_key` configuration on load is intended only for the v5.0 upgrade window.

### Networking: public and intra-cluster networks must be separate.

AIS supports three logical networks:

* **Public network**: user-facing API, CLI/SDK traffic, metrics, and external tooling.
* **Intra-cluster control network**: health, metasync, membership, xaction control, and internal control-plane broadcast.
* **Intra-cluster data network**: object movement between AIS nodes, including target-to-target GET/PUT, global rebalance, resilver, get-batch streams, and other data-mover paths.

These are **logical roles**. They may map to separate physical networks, or they may share physical infrastructure, depending on deployment size and performance requirements.

Starting with v5.0, AIS requires the public network to be separated from intra-cluster networking at the HTTP listener level.

In practical terms:

* the public endpoint must not be the same listener as intra-control or intra-data;
* intra-control and intra-data may share the same listener;
* public multi-homing is allowed, but all public addresses remain public-only;
* physical network isolation is strongly recommended for production, but is not required by this rule.

The mandatory rule is about **HTTP endpoint separation**, not necessarily physical NIC separation. A deployment may use fewer physical networks than logical roles, as long as the public listener is distinct from the intra-cluster listener.

**Example: minimal two-listener deployment**

```json
"host_net": {
    "hostname": "10.50.56.200",
    "hostname_intra_control": "10.50.56.205",
    "hostname_intra_data": "10.50.56.205",
    "port": "51081",
    "port_intra_control": "51081",
    "port_intra_data": "51081"
}
```

In this configuration:

* public traffic uses `10.50.56.200:51081`;
* intra-control and intra-data share `10.50.56.205:51081`;
* AIS runs two HTTP listeners and two HTTP servers: one public, one intra-cluster.

> Note: collapsing intra-nets is not recommended for the usual reasons that boil down to different (maybe even opposite) requirements:
> control traffic is low volume and highly latency-sensitive, whereas intra-cluster data requires high bandwidth, large Tx/Rx buffers, Jumbo frames, etc.

### Direct target access is disabled when AuthN or intra-cluster security is enabled

AIS clients should use proxy endpoints for user-facing API access.

Historically, some deployments allowed users or automation to send object requests directly to target public endpoints, bypassing proxy mediation. Starting with v5.0, direct target access is rejected when either of the following is configured:

* AuthN is enabled;
* `auth.intra_cluster.enabled` is set.

> AuthN and intra-cluster request protection secure different boundaries: identity at the public perimeter, and integrity of internal node-to-node traffic.
> But both rely on the same operational invariant: user traffic must pass through a proxy.

For that precise reason, enabling either security knob disables direct user access to target endpoints.

This applies to user-initiated requests that arrive directly at a target public listener. Proxy-mediated requests, including proxy-to-target redirects, continue to work and are verified internally.

This behavior is intentional. AuthN and intra-cluster request protection guard different parts of the system, but both rely on proxy mediation of user traffic. When either security mode is requested, AIS does not allow clients to bypass the proxy and talk directly to targets.

Operators should update clients, scripts, load balancers, and automation to use proxy endpoints (see `AIS_ENDPOINT`).

Do not point user-facing clients at target public endpoints when AuthN or intra-cluster request protection is enabled.

In v5.0, `auth.intra_cluster.enabled` does not re-enable the removed legacy CSK/HMAC mechanism. It may still affect request-path policy, including rejecting direct target access, as part of the transition toward per-node signing in later v5.x releases.
