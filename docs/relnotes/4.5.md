# AIStore 4.5 Release Notes

AIStore **4.5** is a focused release on three major areas: global rebalance scalability, indexed archive access via the new shard index, and continued hardening of the get-batch (`x-moss`) data path. The release follows AIStore 4.4 and includes roughly five weeks of development across about 110 commits.

The rebalance work is the headline. AIStore removes the legacy per-object ACK machinery that had become a scalability ceiling for large rebalances, replaces the cleanup behavior that depended on that machinery with a new explicit cleanup mode, and hardens the lifecycle around data movers, transport endpoints, stage transitions, and peer status queries.

Remote list-objects is another practical driver for this release. Multi-target remote-bucket listing (flow) now starts the (N-1) targets _before_ the designated (randomly selected) target begins backend listing and page distribution. The same area also gets flow-control cleanup, corrected page accounting, and a dedicated CLI job view for remote-list xactions.

The shard index is a new experimental subsystem for indexed extraction from TAR shards. It lets GET and GetBatch resolve `archpath` lookups using a persisted index instead of scanning the full archive. The 4.5 implementation includes the index format and codec, persistence in a system bucket, a bucket-scoped indexing xaction, CLI support, read-path integration, tests, and a micro-benchmark.

Get-batch continues to mature as a high-throughput ML/data path. The flow is reordered to reduce proxy memory pressure, work-item cancellation now propagates across senders, admission control is stricter under load, and streaming responses use `http.ResponseController` with explicit write deadlines.

Authentication adds support for externally provisioned RSA signing keys, JWKS refresh on cache miss for rotated key pairs, configurable maximum token age, and a more general signing-key configuration. Intra-cluster request validation is also tightened: spoofed caller headers are rejected on the public network, and internal-network checks require a validated Smap entry.

CLI and observability gain several user-visible improvements: dynamic `ais show cluster cpu` and `ais show cluster memory` views, a new `ais performance intra-data` view, shard-index commands, better rebalance rendering including cleanup mode, and improved help for force-join / split-brain recovery workflows.

This release preserves the existing JSON API surface. The few additive API fields, configurable behavior changes, and operational migration notes are summarized in [Upgrade Notes](#upgrade-notes).

<a name="global-rebalance"></a>

## Global Rebalance

AIStore 4.5 delivers the largest rebalance update in several release cycles. The main theme is replacing per-object state with stage-level coordination and explicit cleanup semantics.

### Per-object ACKs removed

The legacy `lomAck` path is removed. This includes ACK messages, sender-side maps of unacknowledged objects, retransmit/wait loops, and the ACK-driven lazy-delete path (`lazydel`).

The replacement model is stage-based:

* rebalance traversal sends objects without keeping per-object ACK state on the sender side;
* a post-traverse barrier replaces the old wait-for-ACK drain;
* transport headers carry a compact opaque payload with the rebalance generation so receivers can reject cross-generation late arrivals;
* rebalance totals shown by the CLI are based on actual transmitted objects and bytes (`OutObjs` / `OutBytes`).

The practical result is lower memory pressure and simpler lifecycle behavior during large rebalances.

### Cleanup mode

Removing per-object ACKs also removed the old incidental mechanism that trimmed misplaced source copies after migration. AIStore 4.5 adds an explicit replacement: **rebalance cleanup mode**.

Cleanup mode walks local mountpaths, identifies misplaced object copies using Smap HRW ownership, and removes a misplaced copy only after verifying that the canonical copy exists in the expected location with matching object identity. Identity checks include size and checksum, and use version / ETag when available.

CLI:

```console
$ ais start rebalance --cleanup
$ ais start rebalance --cleanup --force
```

By default, cleanup mode keeps diverged copies. With `--force`, it can also remove copies that differ from the canonical peer version. This is an advanced operator option.

Cleanup mode is intentionally distinct from regular rebalance:

* it has its own preflight checks;
* it refuses to start while rebalance or resilver is active;
* it requires at least two active targets;
* it bypasses `config.Rebalance.Enabled`;
* it uses no data mover, no streams, and no GFN;
* it skips EC-enabled buckets and busy objects.

`ais show job` and `ais show rebalance` render cleanup-mode runs with a dedicated view that reports removed objects and bytes rather than migration TX/RX counters.

### Lifecycle and transport hardening

Several adjacent changes harden rebalance startup, shutdown, and preemption:

* each rebalance run now constructs a fresh data mover rather than renewing or reusing the previous one;
* duplicate transport endpoint handling is made safer after abort/preempt paths;
* `RebStatus` peer queries are lock-free;
* the `fini` path narrows its mutex section so stream close/unregister does not stall `?rbs=true` probes;
* `SameTargets()` now compares both target membership and rebalance-relevant transition flags, and is used consistently across preempt, renew, and xaction sentinel checks;
* `stageReached()` correctly treats the current stage as reached;
* rebalance CtlMsg now includes stage timing and final status information that remains visible via `ais show job --all`.

### Stream bundles pin Smap at construction

Stream bundles are no longer Smap listeners. The xaction layer owns membership policy, and `transport.Extra.Smap` carries the Smap used to construct a bundle. Listener-driven resync paths are removed, and `DM.Smap()` returns the pinned Smap.

## Shard Index

> Experimental in 4.5. The on-disk format, xaction semantics, and CLI surface may change in subsequent releases.

The shard index is a new subsystem for indexed archive extraction. It is primarily intended for ML and data-lake workloads that repeatedly fetch named samples from large TAR shards.

When a client requests a file inside a TAR shard via `archpath`, AIStore can now resolve the lookup through a persisted index instead of scanning the archive. This fast path is integrated into both GET and GetBatch.

The 4.5 implementation includes:

* a compact TAR shard index format;
* binary `Pack` / `Unpack` support;
* source checksum and size embedded into the index to detect stale indexes after re-upload;
* persistence as objects in the `ais://.sys-shardidx` system bucket;
* a persistent LOM flag for indexed shards;
* a bucket-scoped shard-index xaction built on `BckJogRunner`;
* GET and GetBatch `archpath` fast-path integration;
* CLI support and e2e tests;
* a micro-benchmark for indexed archive extraction.

Index loading is transparent. If a valid index exists, AIS uses it to seek directly to the requested entry. If no index exists, or the index is stale or invalid, the read path falls back to existing archive traversal behavior.

CLI:

```console
$ ais shard-index ais://my-shards
$ ais show shard-index ais://my-shards/object.tar
```

<a name="get-batch-x-moss"></a>

## Get-Batch (`x-moss`)

GetBatch continues to evolve as a high-throughput data path for ML workloads. The 4.5 work focuses on reducing proxy memory pressure, bounding wasted sender work, tightening admission control, and improving streaming response reliability.

### Redirect before sender fan-out

The flow is reordered so the client redirect to the designated target happens before sender fan-out. Sender fan-out then runs asynchronously after redirect.

Previously, the proxy could hold assembly-related state while waiting for all sender startup paths. The new ordering reduces proxy-side memory pressure and better aligns ownership with the designated target.

### Work-item cancellation

A new `OpcodeAbortWI` over the out-of-band target-to-target control path lets a designated target cancel an already-failed work item across senders. This bounds wasted sender work under error and overload conditions.

The cancellation path includes sender-side negative caching of aborted work IDs and periodic cleanup of that cache.

### Admission control

GetBatch admission control now accounts for pending work items and system load more directly. Critical memory or disk pressure returns HTTP 429. High memory pressure and high pending work levels can either throttle or reject requests depending on severity.

This preserves throughput in normal conditions while making overload behavior more predictable.

### Streaming responses

Streaming-mode GetBatch responses now use `http.ResponseController`. AIStore sets explicit write deadlines, flushes headers before the first body write, and coalesces deadline refreshes and flushes.

This improves failure detection for slow or disconnected clients and avoids relying on implicit `net/http` buffering behavior.

### Phase-3 startup acknowledgment

Senders now acknowledge phase-3 startup to the designated target over the out-of-band T2T control path. The designated target tracks expected senders per work item and fails assembly if expected senders do not acknowledge startup within the startup window.

The data path still uses SDM for object transfer; the startup notification itself is control-plane traffic.

### CtlMsg and logging

GetBatch CtlMsg output is clarified and scoped, with job-level and node-level counters separated. Per-work-item CtlMsg is available on assemble errors and in super-verbose mode. xaction IDs are now prefixed with `gbt-`, and several hot-path logs are rate-limited or moved behind verbose/super-verbose logging.

## Intra-Cluster Control Plane and Security

AIStore 4.5 continues separating data-plane traffic from xaction control traffic and tightens validation of requests that claim to be intra-cluster.

### Target-to-target xaction control path

A generic target-to-target xaction control path is added under:

```text
POST /v1/xactions/t2tctrl/{kind}/{xid}/{wid}/{opcode}
```

It runs over `cmn.NetIntraControl` and is used by GetBatch for sender-started notifications and abort propagation. This removes the previous `transport.OpcMossStarted` special case and keeps GetBatch control messages off the SDM data path.

### Intra-cluster request validation

The request-validation changes are security-oriented:

* intra-cluster caller headers are rejected when received on the public network;
* self-joining nodes may use public networking, but the sensitive caller headers are stripped;
* internal-network checks require a validated current-Smap entry;
* `/reverse`, `/cluster`, and `/daemon` each get explicit public and intra-control entry points;
* public entry points perform access checks before dispatching;
* internal entry points keep using internal paths.

These changes reduce the chance that an external client can spoof intra-cluster identity or bypass public access checks.

### Internal HTTP and error handling

Internal HTTP helpers are normalized so callers control `Content-Type` explicitly, with JSON remaining the default when unset. Binary xaction-control payloads use `application/octet-stream`.

`ErrHTTP` handling is reinforced across proxy/target boundaries. In particular, forwarded HTTP errors preserve their status codes, which is important for callers that retry on specific statuses such as 429.

### List-objects flow-control and accounting

Remote-bucket list-objects gets a small startup-ordering fix: non-designated targets are started before the designated target begins backend listing and page distribution. This is a temporary grace-period workaround until the flow gets a proper prepare phase.

The list-objects flow-control code is also cleaned up, and remote-list page accounting is made consistent so list page receives are not double-counted between DM and LSO. `ais show job` uses a dedicated remote-list template with TX/RX page counters.

## AuthN

Authentication gets several changes around RSA signing keys, JWKS, and token validation.

RSA signing configuration is restructured under `auth.rsa.{bits,mode}`. The new `mode: "external"` setting indicates that the RSA signing key is managed outside the AuthN process, which is useful for multi-replica deployments, secret managers, and HSM-backed key management.

In external mode:

* a missing key file at startup is fatal;
* AuthN does not auto-generate the key;
* `POST /v1/auth/keys/rotate` returns 400;
* JWKS is built from the key file during initialization;
* live reload still requires restart.

JWKS lookup now supports rotated key pairs better: on key-cache miss, AuthN refreshes from the issuer before failing validation. A new `max_token_age` config option decouples JWKS cache age from token expiration.

The older environment-controlled signing-mode mechanism is removed in favor of the explicit config model.

## Stats and Observability

### Msgpack-encoded stats path

`stats.NodeStatus` and related control-plane structures can now be encoded as msgpack. JSON remains the default for clients that do not request msgpack.

Native client paths that benefit from lower allocation and faster decoding request msgpack with:

```http
Accept: application/msgpack
```

Server responses identify msgpack payloads with the same content type. Helper code centralizes Accept-header parsing and client-side decoding.

### Structured memory and process stats

`apc.MemCPUInfo` gains structured memory and process-stat fields. Existing flat fields remain, and the new fields are optional.

The CLI uses the richer structure to render the new `ais show cluster cpu` and `ais show cluster memory` views.

### GET-not-found stats feature flag

GET(object) 404s are no longer counted in error stats by default. This avoids inflating error counters for workloads where object-not-found is expected or common.

Operators that prefer the previous behavior can enable the `Count-Object-NotFound-Stats` feature flag at cluster or bucket scope.

### Cgroup v2 memory pressure

The cgroup v2 actual-free estimate is refined to use total file-backed cache minus dirty/writeback pages, rather than relying on `inactive_file`. This improves memory-pressure signaling in containerized deployments.

### Intra-data performance view

A new `ais performance intra-data` view reports data-only intra-cluster transport counters: per-target RX/TX objects, bytes, throughput, and average object size, with cluster-wide totals and an idle indicator.

Transport and DM counters now count data PDUs only, avoiding control-message noise in data-path observability.

## Blob Downloader and Prefetch

The blob downloader gains a per-chunk backend read timeout and better abort behavior.

`BlobMsg.ChunkReadTimeout` is a new optional field. A zero value selects the default. The GET path can also pass the timeout via `apc.HdrBlobReadTimeout`.

Per-range reads now derive their context from the xaction context, so aborting the xaction cancels in-flight backend readers and body copies. Error paths close range readers consistently.

The implementation also revises resource-pressure throttling. Under critical pressure, the blob downloader can reject with HTTP 429.

Prefetch can now pass blob-downloader worker and chunk-size settings. Worker count scales with object size and is capped per job.

CLI:

```console
$ ais prefetch ... --blob-chunk-size <size> --blob-num-workers <n>
```

## CLI

### Cluster CPU and memory views

`ais show cluster cpu` and `ais show cluster memory` now provide dynamic per-node views. The tables skip all-zero columns, support node filtering, and work with JSON and no-header output modes.

Examples:

```console
$ ais show cluster cpu
$ ais show cluster memory
$ ais show cluster cpu <NODE_ID>
```

### Intra-data performance

`ais performance intra-data` shows intra-cluster data movement separately from generic stream counters. It reports RX/TX object counts, bytes, throughput, average size, and cluster-wide totals.

### Rebalance rendering

Rebalance output is more specific and easier to correlate across commands:

* `ais start rebalance` reports the started rebalance ID;
* `ais show job` uses a rebalance-specific renderer;
* cleanup-mode rebalances show removed objects/bytes instead of migration TX/RX counters;
* completed rebalance CtlMsg remains available via `ais show job --all`.

### Shard index commands

The CLI adds commands to build and inspect shard indexes:

```console
$ ais shard-index ais://bucket[/prefix]
$ ais show shard-index ais://bucket/object.tar
```

### Force-join / split-brain recovery help

The advanced (force-join clusters) workflow around `ais cluster set-primary --force` gets clearer help, stronger confirmation text, destination URL validation, and more explicit guidance for the follow-up rebalance.

### Help text and templates

Several help-template and rendering details are cleaned up:

* `ais show cluster` help gets inline examples;
* `command [arguments...]` is shown only when a command truly requires a subcommand;
* `NAME` vs `DESCRIPTION` usage is clarified;
* table helpers are consolidated and reused.

<a name="core-config-and-xactions"></a>

## Core, Config, and xactions

### `BckJogRunner`

`BckJog` is extended with an internal worker-pool runner. The runner auto-tunes worker count based on media type and load and gives xactions a shared implementation for bucket-wide jogger-style work.

TCB is migrated first. The shard-index xaction is the second consumer.

### Chunk-size validation

Chunk-size constants are exported, and bucket-config validation now backfills and checks `chunks.chunk_size` even when automatic chunking is disabled. This prevents legacy bucket properties from persisting `ChunkSize == 0` and later causing divide-by-zero paths on oversized PUT or cold GET.

### Workfile basename overflow

Workfile creation now handles borderline-length object names safely. If decorated temporary workfile names would exceed filesystem basename limits, AIS shortens the temporary basename while preserving the final object path.

### Force-join target initialization

Joining targets now create bucket directories during force-join commit before installing the destination BMD. This prevents missing local bucket directories after cross-cluster force-join workflows.

### Space cleanup

Space cleanup skips chunk manifests as a temporary safeguard.

Separately, rebalance cleanup mode is now the explicit cluster-wide mechanism for safely removing misplaced object copies.

### Cloud/backend fixes

AWS multipart upload now handles S3 client creation failure. OCI region header spelling is canonicalized.

<a name="python-sdk-etl-and-aisloader"></a>

## Python SDK, ETL, and aisloader

### Python SDK v1.24.0

The Python SDK release includes several reliability and compatibility fixes:

* `RetryConfig` and `RetryManager` are picklable, unblocking PyTorch `DataLoader` workers under spawn/forkserver-style multiprocessing;
* object reads retry after clean short EOF by resuming from the current byte offset;
* retry logging emits the final exhausted-retry stack once, while preserving concise per-attempt warnings;
* the legacy `aistore/client.py` compatibility re-export is removed;
* missing `__init__.py` files are added to SDK subpackages so generated API docs include the expected modules.

### ETL streaming

ETL streaming paths are tightened across FastAPI, Flask, and HTTPMultiThreadedServer implementations.

Highlights:

* no-FQN FastAPI hpull GET streams through the shared session without buffering the full upstream object;
* no-FQN FastAPI hpush PUT bridges async request streams into sync readers for constant-memory `transform_stream` consumption;
* HTTPMultiThreadedServer hpush PUT uses a bounded reader around `rfile` and drains unread bytes on close to preserve keep-alive correctness;
* one-shot no-FQN PUT bodies skip local retry and rely on AIS-level retry when the server returns the direct-put transient signal;
* transient direct-put bailouts return `503` with `Ais-Etl-Retry-Reason: direct-put-transient`; exhausted-retry cases continue to use `502`.

### Python tests

Test infrastructure is cleaned up to reduce flakes:

* FastAPI ETL tests eager-import `pydantic.v1` on Python 3.9 to avoid an importlib race;
* environment mutations are scoped to avoid leakage between xdist workers;
* sync and async FastAPI ETL tests are split according to how they exercise the server.

### aisloader

`aisloader` adds `-arch.list` to enable read-only archpath-selective GetBatch listing with `-pctput=0`. `-arch.pct > 0` continues to imply archive listing.

<a name="upgrade-notes"></a>

## Upgrade Notes

### AuthN config migration

The top-level `rsa_key_bits` and `externally_provisioned` settings are replaced by the `auth.rsa.{bits,mode}` config block.

Use:

```json
{
  "auth": {
    "rsa": {
      "bits": 2048,
      "mode": "external"
    }
  }
}
```

for externally provisioned RSA keys.

In external mode, a missing key file at startup is fatal and key rotation through `POST /v1/auth/keys/rotate` returns 400. Environment-controlled signing mode is removed in favor of explicit config.

### Additive APIs

The following API changes are additive and backward-compatible for existing JSON clients:

* `BlobMsg.ChunkReadTimeout` — optional per-chunk backend read timeout for blob downloader; zero selects the default;
* `apc.HdrBlobReadTimeout` — optional GET-path header for blob read timeout;
* `apc.MemCPUInfo` gains optional structured memory and process stats;
* `Accept: application/msgpack` requests msgpack for selected stats/control-plane structures; clients that do not request it continue to receive JSON.

### GET-not-found stats

GET(object) 404s are no longer counted in error stats by default.

To preserve the previous behavior, enable the `Count-Object-NotFound-Stats` feature flag at cluster or bucket scope.

Example:

```console
$ ais config cluster set features=Count-Object-NotFound-Stats
```

### Stream bundle API cleanup

Stream bundles no longer listen for Smap changes and no longer support listener-driven resync APIs such as `ManualResync` or `ListenSmapChanged`. Internal callers should pass the intended Smap at construction time and use `DM.Smap()` when they need to inspect the pinned Smap.

### Shard index is experimental

Shard index is available in 4.5 but remains experimental. The index format, xaction behavior, and CLI surface may change in a later release.

### Rebalance cleanup mode

`ais start rebalance --cleanup` is the new explicit cluster-wide path for safely removing misplaced object copies. It is separate from regular rebalance and from local space cleanup.

Use `--force` only when intentionally removing diverged misplaced copies after verifying that this is the desired operator action.
