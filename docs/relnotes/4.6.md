# AIStore 4.6 Release Notes

AIStore **4.6** is a short-cycle reliability and scalability release following 4.5.

The drivers are practical: stabilizing get-batch under severe ML-style stress, replacing a list-objects [grace period](https://github.com/NVIDIA/aistore/releases/tag/v1.4.5#intra-cluster-control-plane) with a proper 2PC protocol, avoiding cgroup-v2-induced broadcast throttling in large clusters, hardening primary-restart bootstrap, and making prefetch and blob-downloader behavior easier to observe and control.

Get-batch is the largest area of work. The release spans several execution domains that must cooperate under load: assembly into the user response writer, remote senders and data transport, housekeeper-driven graceful termination, abandoned work-item cleanup, and user/self-abort paths. The implementation now uses explicit ownership, receive-side pinning, progress timestamps, stable cleanup windows, bounded abort latency, and stricter late-arrival handling.

Remote list-objects (also referred to as `R-flow`) gets a structural fix. The temporary startup-grace workaround from 4.5 is replaced with the proper begin/commit/abort sequence: targets that are about to receive pages prepare first, the (randomly selected) _designated target_ starts paging only after the receive side is ready, and abort has a dedicated cleanup path.

Large-cluster control-plane broadcast is also improved. In containerized deployments using cgroup-v2 CPU quotas, a proxy with a small visible CPU count could collapse one-to-many broadcasts to single-digit parallelism even when targeting dozens of targets. A dedicated cluster broadcast wait group now covers the mostly I/O-bound intra-cluster fanout paths: keepalive, rebalance, generic broadcasts, and xaction target-to-target control.

The intra-cluster control-plane path is restructured as well. Per-call `context.WithTimeout` plumbing is replaced with statically-configured HTTP clients for the two main control-plane timeout classes, eliminating timer registration, context allocation, and `defer cancel` from every broadcast and keepalive probe. A separate fix covers a former primary restarting without a local cluster map: rather than silently assuming the primary role from local configuration alone, the proxy now probes alternative URLs and validates returned node-state-info first.

Prefetch and blob-downloader observability is another release driver. A single prefetch job can spawn many blob-download child jobs. The 4.6 release adds clearer parent/child reporting, per-xaction stats, bucket-keyed Prometheus metrics, and rejection counters so operators can understand progress and throttling behavior without correlating logs manually. Get-batch and list-objects extend their `CtlMsg` output in the same direction. The cross-cutting picture is summarized in [Observability](#observability).

Backward compatibility is preserved. Operational notes are summarized in [Upgrade Notes](#upgrade-notes).

<a name="get-batch"></a>
## Get-Batch

The work in this 4.6 iteration covers several execution domains that are active during a distributed get-batch request: assembly into the user response writer, remote senders and data transport receive paths, housekeeper-driven graceful termination, housekeeper-driven garbage collection of abandoned work-items (i.e., user requests), and abort paths initiated either by the user or internally by the xaction.

### Work-item lifecycle

Work-item ownership is now an explicit 3-way state rather than a single demand flag. Assembly claims a work item before consuming it; abandoned-work GC may clean only unowned work; and receive paths use pin/unpin discipline, so data and control receives cannot accidentally keep touching a work item that is already cleaning up or abandoned.

Abandoned work-items are marked before cleanup and reject new receive admissions. Cleanup waits for a stable receive/assembly quiet window, using last-receive and last-assembly progress timestamps rather than only the original start time - important for large batches that can legitimately run for a long time while still making progress. The default inactivity timeout is adjusted to 90s (the outer fail-stop remains 5m, with a new stable-window inner loop), sized for large-batch workloads with 1G+ objects under contention.

The work-items memory pool is removed in favor of explicit pin/unpin on the receive side. The housekeeper gets a dedicated x-moss maintenance callback, so abandoned-work cleanup no longer depends on the generic [DemandBase](https://github.com/NVIDIA/aistore/blob/main/xact/demand.go) idle/fini path, and each cleanup pass is capped so the housekeeper remains responsive.

### Abort and late-arrival handling

Abort latency is now bounded. An unbounded `wait-group.Wait()` is replaced with a bounded poll, and `drainRx` is extracted with a `brief` mode for fast cleanup paths. Stuck receive-side work items are forced through cleanup via the abandoned path. Late assembly is rejected after x-moss is aborted or done; late `Assemble` calls after abort cleanup are tolerated; internal stopped sentinels are treated as quiet assembly termination.

GFN error handling is corrected as well. When get-batch engages GFN, not-found remains a soft error while other GFN failures are treated as hard errors. Membership changes are detected and enforced during hole-timeout and failed-GFN paths, so a target-set change no longer looks like an ordinary object-level miss.

### Observability and docs

Control messages now include target IDs and use clearer formatting. Documentation is updated to clarify supported scope and known limitations.

<a name="list-objects"></a>
## Remote List-Objects (R-flow)

Multi-target remote-bucket list-objects - the "R-flow" mechanism, one of several distinct list-objects paths in AIS - gets the proper prepare phase that was anticipated in the 4.5 release notes.

> In a multi-target R-flow run, one target is designated to perform the backend listing and distribute pages to the others. In 4.5, DT waited a brief grace period before issuing the backend call and beginning page distribution - a temporary workaround until the flow gained a proper prepare phase.

The workaround is replaced with an explicit two-phase protocol. The begin phase broadcasts startup, renews the x-lso, opens streams, and registers receive paths. The commit phase performs strict x-lso lookup and starts paging. The abort phase performs best-effort lookup and aborts the xaction if present. The previously-reserved feature flag used by the workaround is removed as part of the same cleanup.

The same area gains better runtime observability, extending the dedicated `ais show job` rendering introduced in 4.5. List-objects now reports lightweight per-request statistics: successful pages, returned entries, errors, and average/max page latency. Control messages include the target name and static request details on one selected target, and the implementation retains the startup Smap for stable R-flow execution.

Prometheus-label handling in the page path is also micro-optimized: variable labels are allocated once per xaction rather than per page.

<a name="large-cluster-broadcast-scalability"></a>
## Large-Cluster Broadcast Scalability

Intra-cluster broadcast scalability is improved for containerized deployments that use cgroup-v2 CPU quotas.

Previously, broadcast parallelism was tied to the visible CPU count. In deployments where an AIS proxy was assigned a small CPU quota - for example, 8 cores in a cluster of dozens of targets - one-to-many fanout could collapse to single-digit parallelism. This affected mostly I/O-bound control-plane paths: generic broadcasts, keepalive, rebalance, xaction target-to-target control, and EC metadata requests.

A dedicated cluster broadcast wait group now provides a minimum floor of `128`, while the existing limited wait group is preserved for CPU- or disk-bound local work.

The intent is simple: local CPU-bound parallelism should continue to respect CPU availability, but cluster fanout should not be collapsed merely because the proxy is running under a cgroup-v2 CPU quota.

<a name="intra-cluster-control-plane"></a>
## Intra-Cluster Control Plane

The intra-cluster control-plane is micro-optimized, and the public-vs-intra header-stamping work from the previous release is carried further.

### Static control-plane clients

Per-call `context.WithTimeout` plumbing is replaced with statically configured `http.Client` instances for the two main control-plane timeout classes: `CplaneOperation` (keepalive probes) and `MaxKeepalive` (broadcast behind essentially every control-plane API). The hot path no longer registers a timer, allocates a context, and defers a cancel on every call.

There are now four intra-cluster HTTP clients: three control clients with separate `http.Client.Timeout` values sharing one intra-control transport (single connection pool), and one data client. The control clients are constructed inline rather than through `cmn/client`.

The tradeoff is that `timeout.cplane_operation` and `timeout.max_keepalive` now require a node restart to re-apply. CLI updates to these knobs report restart-required via `ConfigRestartRequired`. See [Upgrade Notes](#upgrade-notes).

### Intra-cluster header stamping

The header-stamping rule is inverted from "stamp unless we know it's public" to "stamp only when we know it's intra-control" - easier to reason about, and a natural complement to the public/intra validation work from the previous release.

Keepalive also filters redundant heard-from updates in `htrun.call`, avoiding unnecessary writes in paths where the information is already known to be stale or irrelevant. The related `TransportArgs.Timeout` field is renamed to `ClientTimeout` for clarity.

<a name="primary-restart-bootstrap"></a>
## Primary Restart (bootstrap)

Proxy startup is hardened for the case where a former primary restarts without a local cluster map.

The failure mode is subtle but operationally disruptive. A restarted former primary may have no current local cluster metadata, while its local configuration still says that `config.primary_url` points to itself. Before 4.6, that node could interpret the configuration as authority to resume the primary role even after the rest of the cluster had already elected or confirmed a replacement primary.

In the best case, no targets join the impostor primary; it fails to form a cluster, shuts down, and is immediately restarted again by Kubernetes. In worse cases, the behavior can confuse bootstrap and recovery workflows.

AIS now treats local primary configuration as something to confirm, not as sufficient proof of authority. Before assuming the primary role, the proxy probes alternative URLs and validates returned node-state-info. If it discovers a valid current primary, it clears the stale local primary assumption and joins normally as a secondary.

To support this, the public `/health` endpoint (used by external watchdogs) can now satisfy bootstrap discovery. A new `QparamPrimaryCii` composes "return node-state-info" and "must be primary," allowing direct confirmation of the alleged primary. The proxy health handler is refactored to parse and validate the three independent query parameters.

See also:

* [Joining a Cluster: Discovery URL](https://github.com/NVIDIA/aistore/blob/main/docs/lifecycle_node.md)

<a name="prefetch-and-blob-downloader"></a>
## Prefetch and Blob Downloader

Observability for prefetch workloads that use blob-downloader internally is significantly improved.

The field motivation is that one prefetch job can spawn many blob-download child jobs. Without explicit parent/child reporting and per-path counters, it can be difficult to tell whether the system is making progress, throttling blob-download admission, or spending most of its time in cold-GET versus blob-download work.

Prefetch now reports per-xaction statistics separating cold-GET and blob-downloader paths. Control messages show configured parameters, completed cold/blob work, pending work, and blob-downloader admission rejects. Prometheus metrics are added for object counts, bytes, cumulative latency, blob-downloader rejects, and object-level prefetch errors.

Blob-downloader control messages also include parent/origin information. Existing callers identify themselves as `GET`, `api-blobdl`, `x-start`, or a parent prefetch xaction - for example:

```text
s3://abc/this-current-blob-name parent:prefetch[MSXf3OJpU] chunk-size:4MiB, workers:8
```

The blob-downloader also preserves the full object name at startup for later control-message rendering, avoiding lifetime assumptions about caller-provided LOMs.

<a name="space-cleanup"></a>
## Space Cleanup

Space cleanup - the storage cleanup job - is extended to remove stale internal storage entries during the regular cleanup walk, and gets a correctness fix for chunked objects.

System buckets are now routed through dedicated cleanup, and stale shard-index and native bucket inventory entries are removed using source-object liveness checks. System-bucket removals are tracked separately and covered by tests.

Cleanup handling for chunk manifests is also fixed. The cleanup walker now consults the `lmeta` xattr and the `lmflChunk` flag to decide whether to keep or remove chunk metadata. This fixes a silent corruption risk for chunked objects whose names contain dots - `.tar`, `.json`, and similar - which could previously be misidentified as invalid chunk manifests and deleted.

Mountpath walking is refactored so callbacks own LOM loading and lifecycle decisions. The jogger no longer pre-loads or filters copies implicitly; callers (`tcb`, `makencopies`, `nsumm`, `bencodex`, `lom_warmup`, `rechunk`, `shard_idx`) load and handle not-exist cases according to their own semantics.

<a name="shard-index-cleanup"></a>
## Shard Index Cleanup

Shard-index cleanup is added for the case when a shard object is deleted.

When an object with the shard-index flag is removed, the associated object is now removed from the internal `ais://.sys-shardidx` bucket as well. `LOM.RemoveObj` is refactored to clean secondary storage in one path, including chunks, copies, and shard-index entries.

Shard-index lookup remains transparent to clients. If an index is absent, the system can still fall back to sequential archive traversal. The important consistency rule is that when object metadata says a shard index exists, the corresponding index object must exist as well.

<a name="xaction-go-api"></a>
## Xaction Go API

The xaction descriptor gains an explicit IC reporting mode, and the Go xaction wait/status APIs become descriptor-aware.

The descriptor table now distinguishes xactions that report terminal status through IC, xactions that report progress through IC, and xactions that do not report through IC. Go API helpers validate that distinction at the client boundary. IC status/wait helpers fail fast for non-IC xaction kinds (returning the new `cmn.ErrXactNonIC`) instead of silently treating IC as a generic wait mechanism.

> A new [helper](https://github.com/NVIDIA/aistore/blob/main/xact/api_name.go#L50-L78) unifies kind/UUID format validation across xaction API.

A new generic [api.WaitForXaction](https://github.com/NVIDIA/aistore/blob/main/api/xaction.go) dispatches according to descriptor metadata, preserves the full `ArgsMsg` selector, and can wait for idle for xaction kinds that define idling semantics. Tests and tools that used IC wait as a generic wait helper are updated to use descriptor-aware or snaps-based waits (for `PutCopies`, `ECPut`, and `Resilver`).

The xaction package [README.md](https://github.com/NVIDIA/aistore/blob/main/xact/README.md) is revised and extended to cover the descriptor table, ICMode, snaps-based status, IC status, waiting semantics, idling, and rebalance/resilver coexistence.

Python SDK alignment with the descriptor-aware semantics is planned for a follow-up.

<a name="etl"></a>
## ETL

Several ETL improvements landed in this cycle.

The Flask no-FQN hpush PUT path now streams request bodies through Werkzeug `request.stream`, drains unread bytes on close to keep keep-alive sockets clean, and extends the transient direct-put retry contract to Flask one-shot PUT bail cases. Upstream GET responses are wrapped so `close()` releases urllib3 connections, and upstream HTTP statuses are forwarded as-is.

On the AIS side, ETL hpush handling honors `503` plus `Ais-Etl-Retry-Reason: direct-put-transient` from an ETL pod. That combination is treated as a soft error and the PUT is replayed from a replayable LOM-backed source (unlike the one-shot direct-put body the ETL bailed on). Other 5xx responses, and 503 responses without the retry-reason header, retain their existing behavior.

Flask streaming lifecycle coverage and streaming retry tests are also expanded.

<a name="observability"></a>
## Observability

Several pieces of the release improve day-to-day observability for operators. The individual changes appear in their respective feature sections; this section summarizes the cross-cutting story.

### Prometheus

Prefetch gains its own per-target, bucket-keyed Prometheus metrics for the first time:

```text
prefetch.cold.n              objects completed via cold-GET
prefetch.cold.size           --/-- bytes
prefetch.cold.ns.total       --/-- cumulative latency
prefetch.blob.n              objects completed via blob-downloader
prefetch.blob.size           --/-- bytes
prefetch.blob.rejected.n     blob-downloader admission rejects (TooManyRequests)
err.prefetch.n               object-level prefetch errors
```

These distinguish cold-GET and blob-downloader paths, surface admission-control rejects directly (rather than requiring log correlation), and separate object-level errors from xaction-level state.

List-objects also micro-optimizes Prometheus variable-label handling on the page path: labels are now allocated once per xaction rather than per page.

### Per-xaction CtlMsg

Several long-running xactions extended their `CtlMsg` implementations with detailed lifetime and per-job statistics. `CtlMsg` is rendered through the CLI (most directly via `ais show job`), so the improvements are visible without additional tooling.

**Get-batch** `CtlMsg` now leads with `(reqs/objs/files)`, omits empty `job:[]` sections, and includes the target ID. This makes per-target progress legible in multi-target output.

**Remote list-objects (R-flow)** extends the dedicated `ais show job` rendering introduced in 4.5. `CtlMsg` now carries lightweight per-request runtime stats: successful pages, returned entries, errors, and avg/max page latency. Control messages are prefixed with the target name and include static request details on one selected target, so a representative request signature is visible without repeating it across every target's output.

**Prefetch** `CtlMsg` is rewritten to show configured parameters, completed cold/blob work, pending work, and blob-downloader admission rejects:

```text
t[sid]: cfg<user-params> job:[cold:(N,Size) blob:(N,Size rejected:N) pending:(N,Size)]
```

**Blob-downloader** `CtlMsg` adds parent/origin attribution so child jobs are no longer anonymous. Existing callers identify as `GET`, `api-blobdl`, `x-start`, or a parent prefetch xaction:

```text
s3://abc/this-current-blob-name parent:prefetch[MSXf3OJpU] chunk-size:4MiB, workers:8
```

<a name="upgrade-notes"></a>
## Upgrade Notes

### Timeout configuration

Timeout configuration can still be updated at any time, and mixed updates remain valid. Two fields are now restart-only:

```text
timeout.cplane_operation
timeout.max_keepalive
```

Updates to either are accepted but take effect on the next node restart. CLI updates to these two knobs warn that restart is required. Previously, both took effect immediately; the change is a consequence of control-plane optimization work in this release (see [Intra-Cluster Control Plane](#intra-cluster-control-plane)).

### Intra-cluster request handling

Intra-cluster headers are now stamped only for requests known to use the intra-control network.

Deployments using standard AIS nodes and clients should not need changes. Custom integrations that manually construct intra-cluster requests should verify that such requests use the intended intra-control endpoint.

### Xaction Go API

Go xaction IC status/wait helpers now validate whether the selected xaction kind reports through IC, and return `cmn.ErrXactNonIC` when it does not.

Callers using IC wait/status helpers as a generic wait mechanism for non-IC xactions should switch to snaps-based waits or to the new descriptor-aware `WaitForXaction` helper.

Python SDK alignment with the descriptor-aware xaction wait/status semantics is planned for a follow-up.
