# AIStore 4.3 Release Notes

AIStore **4.3** introduces native bucket inventory (NBI) for remote buckets, backed by a new system-bucket namespace. Buckets with millions or tens of millions of objects can now be listed from AIS-managed inventory snapshots instead of traversing the backend on every call.

This release also adds full IPv6 networking support across all three logical networks and the transport streaming layer, with cluster-wide address-family configuration and automatic IPv4 fallback.

4.3 expands authentication and key-management capabilities with JWKS persistence, manual key rotation, standard JWT claims, and a new key-provider abstraction. Client-side multipart download is significantly improved across the Python SDK, Go APIs, and aisloader, with a new multiprocessing-based parallel downloader backed by a shared-memory ring buffer. The Python SDK and ETL toolchain add streaming transform support across multiple server styles and improve retry handling.

Improvements include media-aware worker parallelism, global rebalance hardening, per-bucket GCP credentials, remote-bucket support for `rechunk`, multipart whole-object checksum, and various backend and reliability fixes.

Object HEAD v2, introduced as experimental in 4.2, is now stable.

This release includes over 300 commits since [4.2](https://github.com/NVIDIA/aistore/releases/tag/v1.4.2) and is fully backward compatible.

<a name="native-bucket-inventory-nbi"></a>
## Native Bucket Inventory (NBI)

[Native bucket inventory](https://github.com/NVIDIA/aistore/blob/main/docs/nbi.md) is experimental in 4.3. It gives AIS a built-in way to create and reuse inventory snapshots of **remote buckets**.

NBI currently applies to remote buckets only: cloud buckets (including S3, GCS, Azure, and OCI) and remote AIS buckets.

The goal is straightforward: make listing large remote buckets - those with millions or tens of millions of objects - substantially faster to list. Direct backend listings at that scale can take minutes or longer and repeat the same remote roundtrips on every next-page call. With NBI, AIS builds a cluster-managed inventory snapshot once, then serves subsequent listings from local storage, supporting pagination and prefix filtering without touching the backend.

The implementation includes:

- new CLI commands to create, inspect, and remove inventories
- inventory-backed list-objects via the same distributed flow used for in-cluster `ais://` buckets
- paginated reads and continuation tokens
- prefix filtering over inventory snapshots
- inventory metadata including object count, chunk count, participating targets, and cluster-map version
- support for empty inventory names (lookup by bucket name)
- handling of small buckets where some targets may have nothing to list
- improved cache behavior and page-size handling during listing
- deprecation of the older S3-specific inventory path in favor of a native AIS implementation (see [Upgrade Notes](#upgrade-notes))

> NBI is primarily a latency-reduction tool. Since `ais://` data and metadata are already in the cluster, there is no latency tax to justify a separate inventory layer.
> Still, it remains a valid option for point-in-time consistency (snapshotting) - to be possibly considered in the future.

## System Buckets

System buckets establish a reserved namespace for AIS-managed internal data.

The first system bucket is:

- `ais://.sys-inventory` - stores native bucket inventory snapshots as AIS-managed internal objects

System buckets are created on demand and can be listed and inspected with regular CLI commands. At the same time, they are infrastructure: users are not expected to write into them directly or depend on their internal object layout.

This namespace lays the groundwork for more AIS-managed content in future releases.

<a name="ipv6-networking"></a>
## IPv6 Networking

IPv6 support is now available across all three logical HTTP networks and the transport streaming layer.

It is enabled cluster-wide via the `net.use_ipv6` configuration knob. In production, nodes are typically configured with explicit addresses. When addresses are not configured, each node attempts to discover usable IPv6 addresses and, if none are available, falls back to IPv4. Either way, the effective IP family is applied consistently for the lifetime of the process.

> Setting `net.use_ipv6` in the cluster configuration is a cluster-wide address-family preference, not a dual-stack mode. Once the effective family is determined at startup, AIS avoids dual-stack dialing entirely, with no fallback probing on each connection. This keeps connection setup predictable and eliminates the latency overhead that dual-stack resolution can introduce at scale.

Under the hood, this required moving address-family resolution earlier in startup, before listeners and clients are fully brought up. That keeps listeners, intra-cluster HTTP clients, and related transport paths aligned with the same runtime networking choice, and avoids mixed-family failures such as binding on IPv4 while still attempting IPv6 connections elsewhere. This work also extends local playground and CI coverage, and is documented in [docs/networking.md](https://github.com/NVIDIA/aistore/blob/main/docs/networking.md).

> A reminder that AIS separates traffic across **three logical networks**: a user-facing public network, and two intra-cluster networks for control and data. Each can be placed on a dedicated NIC or VLAN. The most impactful use is isolating the intra-cluster data network - which carries global rebalance, get-batch, and other bulk data movement - from the public network serving foreground I/O. In fact, on loaded clusters, there is also strong motivation to isolate the latency-sensitive (small-bandwidth) control plane from both of the above. Logical network separation is a designed-in capability that pays off most at high drive counts and high utilization, and is worth configuring explicitly rather than relying on shared bandwidth of 100GbE links.

<a name="multi-part-download"></a>
## Multi-Part Download (MPD)

Multi-part download (MPD), also referred to as *parallel download*, sees substantial improvements in 4.3 across the Python SDK, Go client APIs, and [`aisloader`](/docs/aisloader.md).

> For a detailed walkthrough of the architecture, benchmarks, and PyTorch integration,
> see the companion blog post: [Parallel Download: 9x Lower Latency for Large-Object Reads](https://aistore.nvidia.com/blog/2026/03/25/parallel-download).

On the [Python side](https://github.com/NVIDIA/aistore/tree/main/python/aistore/sdk/obj/content_iterator), the parallel downloader has been reworked to use multiprocessing together with shared-memory buffers. This replaces the earlier thread-based design with a model that better matches the underlying I/O pattern: worker processes perform range reads in parallel and write directly into shared memory.

> This currently requires the `fork` start method and is therefore Linux-only.

Two parallel download modes are now supported:

**Streaming (`create_iter`)** keeps a fixed number of range reads in flight and yields data incrementally via a shared-memory ring buffer.

**Bulk (`read_all`)** uses a full-object shared-memory buffer (`ParallelBuffer`). Workers write directly into their assigned byte ranges - thus reducing the copy count and associated (CPU, memory) overhead.
The caller receives the completed buffer without the extra copy previously required on the return path. The caller is responsible for releasing the buffer via `close()` or by using the context manager protocol.

This is especially relevant for large-object workloads, where removing the extra assembly step reduces overhead and makes parallel full-object reads more efficient.

For PyTorch users, 4.3 also adds [`AISParallelMapDataset`](https://github.com/NVIDIA/aistore/tree/main/python/aistore/pytorch), a new map-style dataset for parallel large-object downloads. It wraps `read_all()` with `num_workers` and returns `(object_name, ParallelBuffer)` from `__getitem__`, leaving buffer consumption and cleanup to the collate function or training loop.

> `AISParallelMapDataset` is currently **experimental.** Further optimization - including memory footprint - is planned in subsequent releases.

On the Go side, [`MultipartDownloadStream`](https://github.com/NVIDIA/aistore/blob/main/api/mpt.go) now returns object attributes together with the stream reader. This makes it easier for callers to validate checksums and reuse metadata discovered during the initial HEAD request.

Benchmarking support has expanded as well. [`aisloader`](/docs/aisloader.md) now supports `--mpdstream` as a GET workload variant, with configurable chunk size and worker count, optional checksum verification, and separate statistics for multipart-download operations.

Highlights include:

- multiprocessing-based parallel download in the [Python SDK](https://github.com/NVIDIA/aistore/tree/main/python/aistore/sdk)
- shared-memory bulk reads via `ParallelBuffer`, avoiding the extra copy previously needed by parallel `read_all()`
- new [`AISParallelMapDataset`](https://github.com/NVIDIA/aistore/tree/main/python/aistore/pytorch) for PyTorch-based large-object training and data-loading workflows
- Go [`MultipartDownloadStream`](https://github.com/NVIDIA/aistore/blob/main/api/mpt.go) returning object attributes together with the stream
- new [`aisloader`](/docs/aisloader.md) MPD workload mode and reporting

<a name="multi-part-upload-whole-object-checksum"></a>
## Multi-Part Upload: Whole-Object Checksum

During multipart upload, parts may arrive out of order. Because computing a whole-object checksum previously required re-reading the assembled payload, AIS did not compute whole-object checksums for cloud buckets. In 4.3, AIS computes whole-object CRC32C without rereading data: each part is checksummed as it arrives, and once all parts are present AIS combines the per-part CRC32C values algebraically, regardless of upload order.

Whole-object checksum is now enabled unconditionally. When all parts are present, the whole-object CRC32C is derived from the per-part values using [cos.CRC32CCombine](https://github.com/NVIDIA/aistore/blob/main/cmn/cos/cksum.go) - a GF(2) matrix exponentiation over the Castagnoli polynomial that runs in O(log N) in the length of each part, with no I/O. Parts may arrive in any order.

Additionally, when parts do arrive in order, AIS also computes bucket-configured streaming checksum and stores it as part of object metadata.

## Authentication and Key Management

Authentication and authorization capabilities underwent substantial operational hardening.

The release adds persistent JWKS state, manual key rotation, and public-key retrieval, while also improving configuration validation, token-signing abstractions, retry behavior, and operational diagnostics.

It also completes the transition toward standard JWT claim semantics. The previously deprecated custom username and expires claims are now removed. Issued AuthN tokens now additionally carry standard `iss`, `iat`, and `aud` claims.

From an operational perspective, the main changes include:

- previously deprecated custom `username` and `expires` JWT claims removed (see [Upgrade Notes](#upgrade-notes))
- issued AuthN JWTs now include standard `iss`, `iat`, and `aud` claims
- JWKS persistence to the AuthN database
- public-key retrieval API
- manual key-rotation API with JWKS accumulation (old keys retained for token validation)
- new `KeyProvider` / `ServerKeyProvider` interfaces replacing the prior ad-hoc signing dispatch
- improved validation and defaults for AuthN configuration
- clearer error messages for invalid configuration
- passphrase-based encryption of RSA private keys
- retry support for selected transient AuthN failures

Together, these changes make AuthN easier to operate, safer to configure, and better suited for long-running production deployments. Run `ais auth login` after upgrade; see [Upgrade Notes](#upgrade-notes).

## Media-Aware Worker Parallelism

AIS now detects backing media characteristics at startup and uses a shared runtime policy to auto-scale default worker counts by [mountpath](https://github.com/NVIDIA/aistore/blob/main/docs/terminology.md#mountpath) count and storage type (HDD, SSD, NVMe), while still throttling under high goroutine, memory, or CPU pressure.

The result is better out-of-the-box parallelism for jobs such as:

- all multi-object operations (prefetch, multi-object copy/transform, evict, and more)
- transform/copy bucket
- blob-download
- global rebalance

## Object HEAD v2

The new API to provide object properties was first included in v4.2.

Introduced as experimental, the API is now stable in 4.3 and is used broadly across CLI and backend flows.

> The original (v1) API serialized all object metadata unconditionally - erasure-coding, n-way mirror, and location fields were always included in every response regardless of whether the caller needed them. With the new API, the fields are only populated and transmitted when explicitly requested, resulting in smaller responses and lower marshaling overhead (for the common case) when callers need only a subset of all supported attributes.

Changes in this area include:

- `ais show object` migration to `HeadObjectV2`
- remote AIS `HeadObj` migration to `HeadObjectV2`
- default property sets for object HEAD v2
- continued cleanup around metadata behavior for cached and remote objects

These changes reduce metadata overhead and align more client and backend paths with the newer API.

<a name="rechunk-remote-bucket-sync"></a>
## Rechunk: Remote Bucket Support and `--sync-remote`

Introduced in 4.1, `rechunk` xaction is a batch job that rewrites existing objects between monolithic and chunked layouts according to current bucket settings - for example:

```console
$ ais bucket props s3://abc chunks
PROPERTY                         VALUE
chunks.checkpoint_every          0
chunks.chunk_size                1GiB
chunks.flags                     0
chunks.max_monolithic_size       1TiB
chunks.objsize_limit             0B (auto-chunking disabled)
```

Previously, `rechunk` operated exclusively on in-cluster objects and always skipped the remote backend. This release extends `rechunk` to support remote buckets, with an explicit [`SyncRemote`](https://github.com/NVIDIA/aistore/blob/main/api/apc/rechunk.go) option controlling whether rechunked objects are written back to remote storage via multipart upload.

The default for remote buckets remains local-only - `rechunk` rewrites the local representation without touching the backend. With `SyncRemote=true` (or `--sync-remote` in the CLI), AIS uploads the rewritten objects back to S3, GCS, Azure, OCI, or remote AIS - making the new layout persistent in remote storage as well.

See also:
* [Rechunk control message](https://github.com/NVIDIA/aistore/blob/main/api/apc/rechunk.go)
* [Go `api.RechunkBucket`](https://github.com/NVIDIA/aistore/blob/main/api/bucket.go)

## Runtime, Backend, and Reliability Fixes

Beyond the larger feature work, 4.3 includes a number of robustness and interoperability improvements across runtime paths and backend integrations.

Notable fixes and updates include:

- improved Go client retry behavior with correct request-body reopening
- intra-cluster TLS x509 certificate reload fix: dedicated self-contained `_x509Client()`/`_x509call()` helpers using `InsecureSkipVerify` and `Transport.Clone()`, replacing the prior approach that used a stale TLS client during broadcast
- AWS chunked-content upload support (`sigv4-streaming`)
- cgroup-aware CPU starvation detection, replacing load-average-based heuristics
- ETL initialization race fix: IC registration deferred until after 2PC begin
- clearer remote AIS backend error reporting with cluster identity prefix

Collectively, these fixes reduce operational "surprises" and improve reliability in long-running deployments.

### Minimal Docker Deployment

The minimal Docker deployment - a quick way to spin up a fully functional single-node AIS cluster (one proxy/gateway, one target) without Kubernetes or a full bare-metal setup - has been redesigned as a proper two-container Docker Compose setup.

The previous implementation ran both proxy and target as backgrounded processes inside a single container, which made independent lifecycle management impossible.

The new [setup](https://github.com/NVIDIA/aistore/tree/main/deploy/prod/docker/compose) gives each role its own container, a shared internal network, healthcheck-gated startup (target waits for proxy to be healthy before joining), and named volumes for persistent storage.

Changes:

- **Podman and Docker support** - auto-detected; override with `COMPOSE="docker compose"`
- **Cloud backends** - AWS credentials via env vars or `~/.aws` directory mount (`AWS_CREDENTIALS_PATH`); GCP via service account JSON mount (`GCP_CREDENTIALS_PATH`); image built with `aws gcp azure` support
- **Makefile targets** - `make up`, `make up-build`, `make down`, `make clean`, `make logs`, `make status`
- **Pre-built image** - `aistorage/cluster-minimal:latest` on DockerHub; `make up` pulls automatically

See [deploy/prod/docker/compose/README.md](https://github.com/NVIDIA/aistore/blob/main/deploy/prod/docker/compose/README.md) for quick start and full reference.

### aisloader: Multipart Download Stream Benchmark

`aisloader` now supports benchmarking [multi-part download](https://github.com/NVIDIA/aistore/blob/main/docs/aisloader.md#multipart-download-stream) via new command line:

| Flag | Default | Description |
|---|---|---|
| `--pctmpdstream` | `0` | Percentage of GET operations using MPD stream |
| `--mpdstream-chunk-size` | `8 MiB` | Chunk size for parallel download |
| `--mpdstream-workers` | `16` | Number of concurrent download workers (`0` = API default) |

MPD stream operations appear as `GET-MPDSTREAM` in interval stats output and are tracked separately from regular GETs and get-batch. Checksum verification is supported via `--verify-hash`.

> Note: `--pctmpdstream` is mutually exclusive with `--pctput` and `--batch-size`.

<a name="upgrade-notes"></a>
## Upgrade Notes

### AuthN: token reissuance and JWT claim changes (clusters with authentication only)

> This section **only applies** to deployments that use [AuthN](https://github.com/NVIDIA/aistore/blob/main/docs/authn.md).

Tokens issued by older AuthN versions are invalid after upgrade to 4.3. Run `ais auth login` after upgrading to obtain a new token (or see `ais auth login --help` for details).

The `aud` claim contains the cluster IDs the token is authorized for. AIS gateways can optionally enforce audience matching via `RequiredClaims.Aud` configuration. Admin tokens receive an `aud` claim set to all registered cluster IDs at the time of issuance. If new clusters are registered after an admin token is issued, the admin must re-authenticate to receive an updated audience.

### Bucket metadata (BMD) changes

Bucket metadata is extended with `extra.gcp` to support per-bucket GCP credentials. At the same time, obsolete legacy `extra.hdfs` fields are removed from the current metadata model. For compatibility, AIStore 4.3 continues to read older persisted metadata that still contains those legacy fields.

### Legacy S3 bucket inventory deprecated

The legacy S3-specific bucket-inventory path is now **deprecated** in favor of [native bucket inventory (NBI)](#native-bucket-inventory-nbi). This includes the older S3 inventory listing flow and related internal compatibility paths. Removal is planned for the next (**4.4**) release. Users relying on S3 inventory should migrate to NBI-based listing and inventory management.

> [`xmeta`: AIS Metadata Tool](https://github.com/NVIDIA/aistore/blob/main/cmd/xmeta/README.md)

> As always, backing up cluster metadata (periodically and/or before upgrade) is recommended. In particular, archive metadata from the node configuration directory and, for storage targets, from all mountpaths. Keep the backup off-node.
