# AIStore 4.3 Release Notes (draft)

AIStore **4.3** introduces native bucket inventory for remote buckets, adds IPv6 networking support, expands authentication and key-management capabilities, and improves operational behavior across global rebalance, cloud backends, Python ETL, and the CLI.

The new [native bucket inventory](https://github.com/NVIDIA/aistore/blob/main/docs/nbi.md) (NBI) subsystem enables AIS to create and reuse cluster-managed snapshots of remote bucket contents, so repeated listings can be served from AIS-managed inventory data instead of traversing the remote backend every time.

AIStore 4.3 also adds full IPv6 and dual-stack networking, multi-credential support for GCP backends, media-aware worker parallelism (NVMe/SSD/HDD detection), and continues to harden runtime paths including global rebalance, retries, certificate reload, and backend interoperability.

On the security side, [AuthN](https://github.com/NVIDIA/aistore/blob/main/docs/authn.md) gains JWKS persistence, manual key rotation, standard JWT claims, and a new key-provider abstraction.

The Python SDK and ETL toolchain add streaming transform support across multiple server styles and improved retry handling.

AIStore 4.3 maintains backward compatibility with [v4.2](https://github.com/NVIDIA/aistore/releases/tag/v4.2) and earlier releases.
Overall, this release improves large-scale remote-bucket usability, strengthens security and backend flexibility, and broadens operational visibility across long-running jobs.

<a name="native-bucket-inventory-nbi"></a>
## Native Bucket Inventory (NBI)

Version 4.3 introduces [native bucket inventory](https://github.com/NVIDIA/aistore/blob/main/docs/nbi.md), a new AIS mechanism for creating and reusing inventory snapshots of **remote buckets**.

NBI is designed for large remote namespaces where repeated list-objects calls may otherwise require expensive backend traversal. Instead of scanning the remote bucket on every request, AIS can build a cluster-managed inventory, store it internally, and use it to serve subsequent listings with pagination and prefix filtering.

NBI currently applies to remote buckets only: cloud buckets (including S3, GCS, Azure, and OCI) and remote AIS buckets. It is not used for native user-created `ais://` buckets.

The initial 4.3 implementation includes:

- new CLI commands to create, inspect, and remove inventories
- inventory-backed list-objects flow
- support for paginated reads and continuation tokens
- prefix filtering over inventory snapshots
- inventory metadata including object count, chunk count, participating targets, and cluster-map version
- handling for small and empty inventories
- improved cache behavior and page-size handling during listing
- deprecation of the older S3-specific inventory path in favor of a native AIS implementation

> Native bucket inventory is introduced as an **experimental** feature in v4.3.

### Upgrade notes

- The legacy S3-specific bucket-inventory path is now **deprecated** in favor of native bucket inventory (NBI).
- This deprecation includes the older S3 inventory listing flow and related internal compatibility paths.
- Removal is planned for the next (4.4) release.
- Users relying on S3 inventory should migrate to NBI-based listing and inventory management.

## System Buckets

AIStore 4.3 introduces the concept of **system buckets**: AIS-managed internal buckets that use a reserved dot-prefixed namespace.

The first system bucket is:

- `ais://.sys-inventory` - stores native bucket inventory snapshots as AIS-managed internal objects

System buckets are created on demand and can be listed and inspected with regular CLI commands. At the same time, they remain internal infrastructure: users are not expected to write into them directly or depend on their internal object layout.

This new namespace establishes a foundation for additional AIS-managed internal content in future releases.

<a name="ipv6-networking"></a>
## IPv6 Networking

AIStore 4.3 adds IPv6 support across all three logical HTTP networks and the transport streaming layer.

IPv6 is enabled cluster-wide via `net.use_ipv6` config knob, which expresses preference rather than a hard requirement: on restart, each node attempts to discover usable IPv6 addresses and, if none are available, falls back to IPv4. The selected effective IP family is then applied consistently for the lifetime of the process.

Under the hood, this required reworking network initialization so address-family resolution happens early, before listeners and clients are fully brought up. That keeps listeners, intra-cluster HTTP clients, and related transport paths aligned with the same runtime networking choice and avoids mixed-family failures such as binding on IPv4 while still attempting IPv6 connections elsewhere. The same work also extends local playground and CI coverage, and is documented in the new [docs/networking.md](https://github.com/NVIDIA/aistore/blob/main/docs/networking.md).

## Global Rebalance

Version 4.3 continues to improve the behavior and scalability of [global rebalance](https://github.com/NVIDIA/aistore/blob/main/docs/rebalance.md), especially under load and in the presence of lazy-delete backlog.

This release revises worker scaling, ACK tracking, and lazy-delete interaction to improve responsiveness and reduce long-tail behavior during rebalance-heavy scenarios.

Improvements include:

- scale-out of rebalance workers
- improved lazy-delete responsiveness and logging
- on-the-wire ACK tracking refinements using terminal values
- improved handling when there are no active targets before or after rebalance start
- larger burst-buffer allowance to tolerate lazy-delete backlog

Operationally, the result is a more predictable rebalance path, with better visibility and fewer cases where lazy-delete pressure interferes with forward progress.

## Multi-Credential GCP Backend

AIStore 4.3 adds **multi-credential support for the GCP backend**, enabling different buckets to use different GCP credentials while preserving the usual AIS bucket model.

This is useful in deployments that access multiple GCP projects, service accounts, or administrative domains from the same AIS cluster.

The implementation extends bucket properties with a dedicated GCP configuration section and updates backend initialization to support both bucket-specific and default credential paths.

> This feature extends BMD (bucket metadata) with a new `extra.gcp` section for per-bucket GCP credentials. The 4.3 metadata model also removes obsolete legacy `extra.hdfs` fields while retaining backward-compatible decoding of older persisted metadata.

Highlights include:

- new `extra.gcp` bucket configuration
- per-bucket GCP credential selection
- lazy initialization of the default GCP session
- accompanying documentation updates

This change brings GCP closer to the multi-account flexibility already expected in mixed-cloud or multi-tenant deployments.

<a name="authentication-and-key-management"></a>
## Authentication and Key Management

Building on earlier AuthN work, version 4.3 expands key management, persistence, configuration handling, and public-key access.

The release adds explicit support for persistent JWKS state, manual key rotation, and public-key retrieval, while also continuing to improve configuration validation, token-signing abstractions, retry behavior, and operational diagnostics.

AIStore 4.3 also completes the transition toward standard JWT semantics. Issued AuthN tokens now include standard `iss`, `iat`, and `aud` claims, and deprecated custom `username` and `expires` claims have been removed in favor of the standard `sub` and `exp` claims.

From the operational perspective, the main changes include:

- issued AuthN JWTs now include standard `iss`, `iat`, and `aud` claims
- deprecated custom `username` and `expires` JWT claims removed (see [Upgrade Notes](#upgrade-notes))
- JWKS persistence to the AuthN database
- public-key retrieval API
- manual key-rotation API with JWKS accumulation (old keys retained for token validation)
- new `KeyProvider` / `ServerKeyProvider` interfaces replacing the old `sigConf` dispatch
- improved validation and defaults for AuthN configuration
- clearer error messages for invalid configuration
- passphrase-based encryption of RSA private keys
- retry support for selected transient AuthN failures

Together, these changes make AuthN easier to operate, safer to configure, and better suited for long-running production deployments.

<a name="media-aware-worker-parallelism"></a>
## Media-Aware Worker Parallelism

AIS now detects backing media characteristics at startup and uses a shared runtime policy to auto-scale default worker counts by [mountpath](https://github.com/NVIDIA/aistore/blob/main/docs/overview.md#mountpath) count and storage type (HDD, SSD, NVMe), while still throttling under high goroutine, memory, or CPU pressure. The result is better out-of-the-box parallelism for jobs such as:

* all multi-object operations (prefetch, multi-object copy/transform, evict, and more)
* transform/copy bucket
* blob-download
* global rebalance

<a name="upgrade-notes"></a>
## Upgrade Notes

### AuthN token reissuance required (clusters with authentication only)

> **This section only applies to deployments that use [AuthN](https://github.com/NVIDIA/aistore/blob/main/docs/authn.md).**

AuthN tokens issued by older versions may become invalid after upgrade to 4.3. AIStore 4.3 removes the deprecated custom `username` and `expires` claims in favor of standard `sub` and `exp`, and issued tokens now include standard `iss`, `iat`, and `aud` claims.

This applies to all tokens, including long-lived admin tokens. Re-login after upgrade to obtain tokens with standard claims.

### Bucket metadata (BMD) changes

Bucket metadata is extended with `extra.gcp` to support per-bucket GCP credentials. At the same time, obsolete legacy `extra.hdfs` fields are removed from the current metadata model. For compatibility, AIStore 4.3 continues to read older persisted metadata that still contains those legacy fields.

### Standard JWT claims

Issued AuthN tokens now include standard `iss` (issuer), `iat` (issued-at), and `aud` (audience) claims. The `aud` claim contains the cluster IDs the token is authorized for. AIS gateways can optionally enforce audience matching via `RequiredClaims.Aud` configuration.

Admin tokens receive `aud` set to all registered cluster IDs at the time of issuance. If new clusters are registered after an admin token is issued, the admin must re-authenticate to receive an updated audience.

> [`xmeta`: AIS Metadata Tool](https://github.com/NVIDIA/aistore/blob/main/cmd/xmeta/README.md)

> As always, backing up cluster metadata (periodically and/or before upgrade) is recommended. In particular, archive metadata from the node configuration directory and, for storage targets, from all mountpaths, and keep the backup off-node.
